# ljyd 交互协议规范质量评估 Skill

基于语雀文档《资源管理流程》（`afahpl/project_doc` → slug `owg3sv8eodzqkcq4`）沉淀的设计规范，
对 ljyd 设备/小程序/智能体平台交互协议及其 Mermaid 流程图做**多维度质量评估**，输出可落地的结构化评分报告与修正建议。

## 适用场景

- 用户提供 Mermaid 流程图（`sequenceDiagram` / `flowchart` / `stateDiagram` 等）要求评审
- 用户要求审查设备接口设计（HTTP / MQTT）的规范性与一致性
- 用户要求评估流程文档的整体质量（逻辑 / 命名 / 格式 / 状态管理 / 场景覆盖）
- 用户给出"流程图 + 接口定义 + 状态/场景描述"组合材料要求综合打分

## 输入模板（建议）

```markdown
评估对象：
- 流程图源码：<Mermaid 代码块，可多个>
- 接口定义：<HTTP 接口列表 / MQTT 接口列表 / OpenAPI 片段>
- 状态/生命周期说明：<可选，状态机或生命周期描述>
- 场景清单：<可选，需支持的业务场景列表>
- 业务背景：<可选，用于判断逻辑合理性与场景全面性>

评估重点：<可指定仅评估部分维度，默认全维度>
对照基线：<可选，默认使用语雀《资源管理流程》规范>
```

若输入不完整，先标注"待确认项"并按可见内容评估；不要为补全格式而臆造业务约束。

## 评估维度与检查清单

### 维度 1：流程逻辑合理性（权重 20%）

验证流程步骤的**连贯性与逻辑闭环**。

检查项：
- **起止完整**：每个流程有明确起点与终止态，不存在悬空步骤或断头路径。
- **步骤连贯**：相邻步骤之间存在因果或时序关系，无逻辑跳跃、无缺失中间环节。
- **参与者一致**：`participant` 定义与消息收发方吻合，无"未定义参与者"出现于消息中。
- **方向正确**：请求 `->>` 与返回 `-->>` 方向符合调用语义；同步/异步标注与业务一致。
- **分支完备**：`alt`/`opt`/`par` 分支覆盖该步骤的全部互斥/可选/并行情况，含明确的结束合并。
- **无死循环**：除显式轮询/重试外，不存在无法退出的环。
- **与业务背景自洽**：流程步骤能支撑所述业务目标，关键业务动作均有体现。

### 维度 2：命名规范合理性（权重 15%）

检查流程中各元素命名的**一致性与专业性**。

检查项：
- **参与者命名**：`participant Id as 显示名`，`Id` 仅字母数字下划线、无空格；中文放 `as` 后；同类系统跨图命名一致（如"管理端"不要一处 `Admin`、一处 `Management`）。
- **消息标签**：简短达意，动宾结构或"名词+动词"；同类操作跨图措辞统一（如统一用"查询"/"下发"，不混用"获取/拉取/取"）。
- **接口/字段命名**：接口路径用小写连字符或驼峰且全篇统一；字段名采用统一风格（项目基线为小驼峰，如 `resourceBundleName`、`parentId`、`sortOrder`）。
- **状态命名**：状态名使用业务语义而非技术态（如"已订阅"优于"state=1"）；同构状态跨图命名一致。
- **无歧义缩写**：缩写需在业内通用或首次出现处注释；避免自创缩写。

### 维度 3：格式规范检查（权重 10%）

确保流程呈现格式**符合文档标准**。

检查项：
- **图类型选择**：交互时序用 `sequenceDiagram`（项目基线首选），状态流转用 `stateDiagram-v2`，整体流转用 `flowchart`；不滥用图类型。
- **代码块规范**：使用 ` ```mermaid ` 围栏；首行为图类型声明；可选 `autonumber` 用于需步骤序号场景。
- **语法合法**：可在 Mermaid 渲染器无错解析；无残留 `flowchart`/`graph TD` 替代 `sequenceDiagram` 的误用。
- **可读性**：单图步骤数适中（建议 ≤ 20 条消息），超长应拆分；标签无超长溢出；注释 `Note` 用于补充而非堆砌。
- **编号与标题**：图前有"X.Y 流程名"层级标题；多图编号连续；与正文接口章节编号呼应。

### 维度 4：设备接口 HTTP/MQTT 双接口一致性（权重 15%）

设备接口须**同时提供 HTTP 与 MQTT 两种通道**，二者业务内容完全一致，仅报文头格式不同。

检查项：
- **双通道并存**：每个设备接口均给出 HTTP 版与 MQTT 版，缺一记为重大缺陷。
- **业务内容一致**：接口名、请求参数（字段名/类型/必填/语义）、响应业务字段（`result` 内结构）在 HTTP 与 MQTT 间**完全一致**。
- **仅报文头差异**：差异应只存在于报文头/封装层：
  - HTTP：`Authorization: Bearer {token}`、`Content-Type: application/json`、环境域名 + 路径、GET Query 拼接。
  - MQTT：Topic 命名规范、QoS、保留消息策略、连接鉴权字段；业务 payload 与 HTTP `result`/请求参数同构。
- **Topic 与路径映射**：MQTT Topic 与 HTTP 路径存在可追溯的对应关系（命名规则一致或附映射表）。
- **响应封装一致**：HTTP 顶层 `msg/status/ok/timestamp/result` 与 MQTT payload 顶层结构一致或明确等价。
- **无业务逻辑漂移**：两通道不出现一方独有的业务字段或独有分支（鉴权/传输层字段除外）。

### 维度 5：接口设计规范性（权重 15%）

评估接口定义的**完整性与合规性**（对齐语雀基线）。

检查项：
- **通用协议头**：声明请求方式（基线 GET）、编码 UTF-8、传输格式（Query 拼接 + JSON 响应）、多环境域名（dev/test/prod）。
- **必传 Header**：`Authorization`、`Content-Type` 齐全且取值说明清晰。
- **通用响应顶层字段**：`msg`/`status`/`ok`/`timestamp`/`result` 齐全；`status=0` 表成功约定一致。
- **公共结构体复用**：跨接口共用结构（如 `CategoryItem`）抽为公共定义并被各接口引用，而非重复定义。
- **单接口完整性**：每个接口含 请求地址 / 请求参数表（参数名/类型/是否必填/说明）/ 响应示例；必填项标注；示例 JSON 与字段表一致。
- **字段说明**：每个字段有类型与语义说明；枚举值列出取值与含义；时间格式注明（如 `yyyy-MM-dd HH:mm:ss`）。
- **错误处理**：非 0 状态码有错误码表或错误描述约定；不遗留"成功响应即全部"的缺口。
- **版本与兼容**：接口路径含版本或模块前缀（如 `/agent-device/...`）；破坏性变更有版本策略。

### 维度 6：生命（状态）管理整齐合理性（权重 10%）

评估资源/设备/流程的**状态或生命周期管理是否整齐、合理**。

检查项：
- **状态集合明确**：列出全部状态取值（如订阅态、上下线态、启用/禁用态），无隐式未命名状态。
- **状态机闭环**：用 `stateDiagram-v2` 或表格表达状态流转；每个状态有可达入边与合理出边；终态明确。
- **流转条件清晰**：每条迁移标注触发条件与动作；无"无因迁移"。
- **整齐性**：同构对象（如多类资源）状态集合与迁移规则一致，不出现同类对象状态定义参差。
- **与流程/接口呼应**：状态变更由流程步骤或接口触发；流程图与状态图能对应，不脱节。
- **无死状态**：不存在无法进入或无法退出的孤立状态（终态除外）。

### 维度 7：场景支持全面性与合理性（权重 15%）

验证所设计流程与接口对业务场景的**覆盖全面且合理**。

检查项：
- **场景清单完整**：覆盖 正常流程 + 异常/失败流程 + 边界场景（空数据、超长、并发、重试、超时、鉴权失败）。
- **场景与流程对应**：每个场景在流程图或接口中可找到支撑路径；无"有场景无路径"。
- **合理取舍**：场景覆盖密度与业务重要性匹配，不过度设计低频边缘场景、也不遗漏高频核心场景。
- **变更/增量场景**：含资源变更查询、增量同步等动态场景（基线含"核心流程（增加资源变更查询）"）。
- **外部/内部区分**：外部资源与内部资源场景分别建模，不混用同一流程（基线区分"外部资源/内部资源"）。
- **可扩展性**：新增场景的接入点清晰（如新增分类、新增资源类型），不需推翻既有流程。

## 评分规则

- 每维度按 0–100 分打分，再按权重加权得总分。
- 等级映射：A(≥90 优秀) / B(80–89 良好) / C(70–79 合格) / D(60–69 待改进) / E(<60 不合格)。
- 单项存在"重大缺陷"（如双接口缺一、状态机有死状态、流程断头）时，该维度直接不超过 C，并在报告中高亮。
- 默认全维度评估；用户指定子集时，权重按指定维度归一化重算。

## Agent 执行步骤

1. **解析输入**：提取流程图源码、接口定义、状态/场景说明；识别缺失项并标注。
2. **基线对齐**：默认以语雀《资源管理流程》规范为对照基线（通用 Header、通用响应字段、`CategoryItem` 公共结构体、`sequenceDiagram` 首选、HTTP 4 接口范式）；用户另指定基线时从其指定。
3. **逐维评估**：按上述 7 维检查清单逐项核验，记录"通过/警告/缺陷"及证据（引用具体图行或字段）。
4. **重大缺陷识别**：扫描双接口缺失、状态死锁、流程断头、字段不一致等硬伤，单独标记。
5. **打分与定级**：按权重加权得总分并映射等级。
6. **输出报告**：按下方模板输出结构化报告，含修正建议（可落地、指明修改位置）。
7. **不臆造**：仅基于可见材料评估；缺材料项标注"待确认"，不扣分但提示补全后复评。

## 对照基线速查（语雀《资源管理流程》要点）

- **图类型**：交互时序首选 `sequenceDiagram`，状态流转用 `stateDiagram-v2`。
- **通用 Header**：`Authorization: Bearer {token}`、`Content-Type: application/json`。
- **通用响应顶层**：`msg`(string, 成功固定 `OK`) / `status`(int, `0`=成功) / `ok`(boolean) / `timestamp`(number 毫秒) / `result`(object 业务载体)。
- **公共结构体**：`CategoryItem`（`id`/`name`/`resourceBundleName`/`parentId`/`level`/`isLeaf`/`sortOrder`/`createTime`/`description`/`icon`/`backgroundImage`/`children`），跨接口复用不重复定义。
- **单接口四要素**：请求地址（含 `GET /agent-device/...`）、Query 请求参数表（参数名/类型/是否必填/说明）、响应参数样例（JSON）、多环境域名。
- **设备接口范式**：HTTP 接口 1–4（获取已订阅分类树 / 获取指定分类直属子分类 / 根据资源包获取资源 / 根据资源 id 获取资源详情）；评估时要求每接口同时给出 MQTT 版，业务内容一致仅报文头不同。
- **场景分层**：区分外部资源 / 内部资源；含资源变更查询增量场景。
