# 产品需求规范

## 1. 需求文档结构

每份 PRD 按以下结构组织：

```
1. 文档信息（版本、作者、日期、变更记录）
2. 背景与目标（为什么要做、解决什么问题、预期收益）
3. 范围边界（做哪些、不做哪些、假设条件）
4. 用户故事（角色 -> 动作 -> 期望结果）
5. 功能详述（按模块拆解，每个功能点描述用户操作与系统响应）
6. 非功能需求（性能、安全、兼容性）
7. 验收标准（可测试、可验证的通过条件）
8. 待确认项与风险
```

## 2. 用户故事格式

采用标准三段式：

```
作为 <角色>，我希望 <功能>，以便 <价值/目标>。
```

### 示例
```
作为 普通用户，我希望 通过手机号+验证码登录，以便 无需记忆密码即可快速登录系统。
```

### 用户故事要求
- 每个故事独立可交付，可单独验收
- 故事粒度适中，不超过 3 天开发量
- 每个故事标注优先级：P0（必须）、P1（应该）、P2（可选）
- 故事之间标注依赖关系

## 3. 功能描述规范

### 描述格式
每个功能点按 **用户动作 → 系统响应 → 验收标准** 组织：

```
功能：手机号验证码登录

用户动作：
1. 用户输入手机号
2. 用户点击"获取验证码"
3. 用户输入收到的验证码
4. 用户点击"登录"

系统响应：
- 手机号格式校验，不合法时提示"请输入正确的手机号"
- 60 秒内同一手机号不可重复发送，按钮倒计时
- 验证码 5 分钟内有效，过期提示重新获取
- 验证码错误提示"验证码错误，请重新输入"
- 登录成功跳转至首页

验收标准：
- [ ] 输入正确手机号和验证码可成功登录
- [ ] 输入错误验证码提示错误信息
- [ ] 60 秒内不可重复获取验证码
- [ ] 过期验证码不可使用
- [ ] 未注册手机号自动注册并登录
```

### 状态覆盖
每个功能必须覆盖以下状态：

| 状态 | 描述 |
|------|------|
| 正常态 | 功能正常使用流程 |
| 空态 | 无数据时的展示与引导 |
| 加载态 | 数据加载中的反馈 |
| 错误态 | 操作失败时的提示与处理 |
| 边界态 | 极限值、特殊字符、并发等 |
| 权限态 | 不同角色的可见性与操作权限 |

## 4. 优先级定义

| 优先级 | 定义 | 判断标准 |
|--------|------|----------|
| P0 | 必须做 | 不做系统无法上线或核心流程不通 |
| P1 | 应该做 | 影响用户体验但可通过替代方案完成 |
| P2 | 可以做 | 锦上添花，不影响核心流程 |

## 5. 范围边界

### 必须明确
- **做哪些**：本次迭代包含的功能范围
- **不做哪些**：明确排除的内容，避免范围蔓延
- **假设条件**：需求成立的前提假设
- **依赖项**：依赖的其他系统、团队、接口

### 示例
```
范围：
- 做：手机号验证码登录、手机号密码登录
- 不做：第三方登录（微信/支付宝）、人脸识别登录

假设：
- 短信服务已接入并可正常使用
- 用户手机号唯一

依赖：
- 短信服务商接口
- 用户中心基础服务
```

## 6. 非功能需求

- **性能**：页面加载 < 2s，接口响应 < 500ms（P95）
- **并发**：支持同时在线用户数、QPS 预估
- **安全**：数据传输加密、敏感信息脱敏、操作审计
- **兼容性**：支持的浏览器、操作系统、屏幕分辨率
- **数据**：数据量预估、增长趋势、存储周期

## 7. 验收标准

- 每条验收标准可测试、可验证
- 使用 `[ ]` 复选框格式，便于验收勾选
- 覆盖正常流程 + 异常流程 + 边界条件
- 验收标准与用户故事一一对应

## 8. 需求评审检查清单

- [ ] 背景与目标是否清晰
- [ ] 用户故事是否完整且可独立交付
- [ ] 功能描述是否覆盖所有状态
- [ ] 优先级是否合理
- [ ] 范围边界是否明确
- [ ] 非功能需求是否考虑
- [ ] 验收标准是否可测试
- [ ] 依赖项是否已确认可行
- [ ] 待确认项是否有负责人和截止时间
