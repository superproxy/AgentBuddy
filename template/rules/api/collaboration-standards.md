# API 协作规范

## 1. 接口设计原则

- 前后端接口由后端主导设计，前端参与评审
- 接口设计先于开发，评审通过后方可开发
- 接口契约一经确认不得单方面修改
- 接口文档是前后端的共同约定，双方共同维护

## 2. 接口文档规范

### 文档格式
使用 OpenAPI 3.0+ 或项目约定的文档格式，每个接口包含：

```yaml
路径: /api/users/{id}
方法: GET
描述: 获取用户详情
请求参数:
  - name: id
    in: path
    type: integer
    required: true
    description: 用户 ID
响应:
  200:
    description: 成功
    body:
      code: 0
      message: "success"
      data:
        id: 1
        name: "张三"
        email: "zhangsan@example.com"
        created_at: "2024-01-01T00:00:00Z"
  404:
    description: 用户不存在
    body:
      code: 40401
      message: "用户不存在"
```

### 文档要求
- 每个字段标注：类型、是否必填、取值范围、示例值
- 枚举字段列出所有可能的值
- 日期时间统一使用 ISO 8601 格式（`2024-01-01T00:00:00Z`）
- 金额单位为分（整数），避免浮点精度问题
- 响应示例覆盖成功和常见失败场景

## 3. 接口版本管理

### 版本策略
- URL 路径版本：`/api/v1/users`、`/api/v2/users`
- 兼容性升级不增加版本号（新增可选字段、新增接口）
- 破坏性变更必须增加版本号（删除字段、修改字段类型/含义、删除接口）
- 旧版本至少保留一个迭代周期后废弃

### 变更流程
1. 后端提出变更，更新接口文档
2. 前端评估影响范围
3. 双方确认变更方案和兼容策略
4. 后端实现并部署
5. 前端适配并上线
6. 确认无流量后废弃旧版本

## 4. 前后端协作流程

### 开发前
1. 后端输出接口文档初稿
2. 前端评审接口字段是否满足页面需求
3. 双方确认后锁定接口契约
4. 后端提供 Mock 接口或 Mock Server

### 开发中
- 前端基于 Mock 数据并行开发
- 后端开发真实接口
- 接口有调整时及时同步文档并通知前端
- 使用接口管理平台（如 YApi/Apifox/Swagger）统一管理

### 联调阶段
1. 后端自测通过后通知前端联调
2. 前端切换 Mock 到真实接口
3. 逐接口验证：正常数据、空数据、错误数据、边界数据
4. 联调问题记录在接口文档或协作平台上

### 提测前
- [ ] 所有接口已联调通过
- [ ] 接口文档与实际实现一致
- [ ] 异常场景已覆盖（网络错误、超时、服务不可用）
- [ ] 接口响应时间满足性能要求

## 5. 接口命名规范

### URL 规范
- 使用小写字母和连字符：`/api/user-orders`
- 资源使用名词复数：`/api/users`
- 资源关系通过路径表达：`/api/users/{id}/orders`
- 避免深层嵌套，最多 3 层：`/api/users/{id}/orders/{orderId}/items`
- 非 CRUD 操作使用子资源：`/api/orders/{id}/cancel`

### 字段命名
- 请求/响应字段使用 camelCase：`userId`、`createdAt`
- 数据库字段使用 snake_case：`user_id`、`created_at`
- 布尔字段使用 `is` 前缀：`isActive`、`hasPermission`
- 时间字段统一后缀：`createdAt`、`updatedAt`、`deletedAt`

## 6. 错误处理规范

### 错误码设计
```
格式：{模块码}{错误类型码}

模块码：
10 - 用户模块
20 - 订单模块
30 - 商品模块
40 - 支付模块
50 - 系统通用

错误类型：
01 - 参数错误
02 - 资源不存在
03 - 权限不足
04 - 业务规则限制
99 - 系统错误

示例：
10001 - 用户参数错误
10002 - 用户不存在
20004 - 订单状态不允许此操作
```

### 错误响应
```json
{
  "code": 10001,
  "message": "用户参数校验失败",
  "details": [
    { "field": "email", "message": "邮箱格式不正确" },
    { "field": "age", "message": "年龄必须在 1-150 之间" }
  ]
}
```

## 7. 接口性能要求

| 接口类型 | P95 响应时间 | P99 响应时间 |
|----------|-------------|-------------|
| 查询单个 | < 200ms | < 500ms |
| 查询列表 | < 500ms | < 1s |
| 创建/更新 | < 500ms | < 1s |
| 批量操作 | < 2s | < 5s |
| 导出 | < 10s | < 30s |

## 8. Mock 规范

- Mock 数据结构与真实接口完全一致
- Mock 数据覆盖：正常数据、空列表、单条数据、大量数据
- Mock 支持模拟：成功、失败、超时、网络错误
- 前端可通过请求头或参数切换 Mock/真实接口
- Mock 不阻塞前端开发，后端接口就绪后平滑切换
