# 前端编码规范

## 1. 组件设计

### 组件边界
- 每个组件职责单一，只做一件事
- 容器组件负责数据获取与状态管理，展示组件负责 UI 渲染
- 组件 Props 使用 TypeScript 接口明确定义类型
- 避免超过 3 层的 Props 嵌套传递，优先使用 Context 或状态管理

### 组件命名
- React 组件文件使用 PascalCase：`UserProfile.tsx`
- Vue 组件文件使用 PascalCase：`UserProfile.vue`
- 页面组件放在 `pages/` 或 `views/` 目录
- 通用组件放在 `components/` 目录
- 业务组件放在对应业务模块目录下

## 2. 页面状态

每个页面/组件必须覆盖以下四类状态：

| 状态 | 要求 |
|------|------|
| Loading | 数据加载中显示骨架屏或加载指示器 |
| Empty | 无数据时显示友好的空状态提示与操作引导 |
| Error | 请求失败时显示错误信息与重试入口 |
| Success | 正常展示数据，处理边界情况 |

## 3. 交互反馈

- 所有异步操作（提交、删除、保存）必须有 loading 状态
- 操作成功/失败后使用 Toast/Message 给出明确反馈
- 危险操作（删除、退出）必须有二次确认
- 表单提交后按钮禁用，防止重复提交

## 4. 接口联调

- 统一使用项目封装的 request 工具，不直接使用 fetch/axios
- 接口请求必须处理异常，使用 try-catch 或 .catch()
- 接口响应数据需做空值校验后再使用
- 不擅自扩展接口参数语义，严格按后端契约调用

## 5. 代码风格

### 命名规范
- 变量/函数：camelCase
- 常量：UPPER_SNAKE_CASE
- 类型/接口：PascalCase
- CSS 类名：kebab-case 或 BEM 命名

### 文件组织
- 每个文件只导出一个主要组件/模块
- 相关的类型定义放在同目录的 `types.ts`
- 工具函数放在 `utils/` 目录

### 代码质量
- 优先使用 const，避免 var
- 使用解构赋值简化代码
- 避免魔法数字，使用命名常量
- 复杂条件判断抽取为命名函数
- 禁止使用 any，必须明确类型

## 6. 性能与可访问性

- 列表渲染使用唯一且稳定的 key
- 大列表使用虚拟滚动或分页
- 图片使用懒加载
- 交互元素提供合适的 aria 属性
- 确保键盘可操作
